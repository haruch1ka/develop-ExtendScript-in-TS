# 開発ルール

## 作業開始の条件

- 作業は、明確にユーザーが指示を与えるまで開始しないこと。

## 開発の流れ

Claude Code とユーザーがプログラム開発をする際は、通常のプログラム開発と同じ流れをたどること。

1. 要件定義
2. 基本設計
3. 詳細設計
4. コーディング
5. テスト
6. 受け入れテスト

---

## 要件定義駆動の原則

- **曖昧な依頼をそのまま実装に渡さない。** まず構造化された仕様書（Markdown）に変換し、その仕様に基づいて実装・レビューを行うこと。
- 「いい感じに作って」ではなく、「この仕様の Must 要件を、この受け入れ条件下で実装する」という形に落とし込んでから着手する。
- 推測や思い込みで仕様を埋めない。不明点はユーザーへの質問で確定させる（後述の質問ルールに従う）。

### 要件定義の 7 ステップ

上記「開発の流れ」の 1.要件定義 は、以下の順で進める。

1. **背景・目的の確認** — 何のために、どんな課題を解決するのかを整理する。
2. **ターゲット・利用主体の整理** — 誰が・どの環境で使うのか（利用者、実行環境、対象データ）を定義する。
3. **機能分類（Must / Should / Could）** — 必要な機能を洗い出し、優先度で分類する。
4. **Must 機能の詳細化** — コアとなる Must 機能のみを深掘りする。
5. **仕様の明示化** — 入出力（I/O）、バリデーション、エラーハンドリング、非機能要件を具体化する。
6. **受け入れ条件の設定** — 各要件に対して「満たせたと判断できる基準」を定める。
7. **Markdown 仕様書の作成** — 上記を構造化した仕様書として出力する。

### Must 要件の詳細化ルール

- **曖昧表現を禁止する。** 例:「適切にエラーを出す」ではなく、以下まで落とし込む。
  - どの条件がエラーになるか
  - 表示するメッセージ内容
  - ユーザーの次のアクション
  - 再試行の可否
  - データ保存（永続化）の有無
- Must 要件では、受け入れ条件・入出力仕様・処理フロー・バリデーション・エラーハンドリング・データ永続化まで具体化する。
- **デフォルトの実装対象は Must のみ。** Should / Could はユーザーが明示的に指示した場合にのみ対象とする。

### 曖昧さの分離（TBD リスト）

- 未確定項目・推測が入る項目は本文に混ぜず、**「TBD リスト」として明示的に分離**する。
- TBD に残った項目は実装対象から除外する。確定してから実装に含めること。

### 仕様 ID による紐付け

- 各機能要件に ID を付与する（例: `F-001`）。異常系・バリデーション要件にも ID を付与する（例: `V-001`）。
- 実装の指示は ID 単位で行う（「F-001 を、その受け入れ条件下で実装する」）。
- レビューは感想ではなく ID の受け入れ条件との照合で行う。
  - ✕「この実装は良いか？」
  - ○「F-001 の受け入れ条件を満たしているか？ 異常系 V-003 は実装されているか？」

### 適用の境界

- **要件定義駆動を適用する:** 新機能、複数ファイルにまたがる実装、複合的な処理、失敗時の振る舞いが重要なもの、今後保守するもの。
- **適用しなくてよい:** typo 修正、軽微なリファクタ、創造的な試行錯誤、使い捨てのラフな試作。

## ドキュメントの作成場所

- 要件定義・基本設計・詳細設計は `docs/plans/${project名}/` に作成する。
- 作成するものの前提を細かく取り決める。

## 計画段階での質問の仕方

- 計画や作りたいものに関する情報はユーザーの頭の中にある。計画の段階で不明な点があるときは、各ファイルを参照せず、ユーザーに質問して計画を詰めること。
- 複数質問がある場合は、順に1つずつ質問すること。
- 1つのメッセージに複数の質問を並べてはならない（箇条書き・番号付きであっても不可）。質問ツールが選択肢2つ以上を要求する等の制約でツールが使えない場合も、この原則を緩めず、平文で1問ずつ質問すること。
- ユーザーからルール違反を指摘された場合は、(1) 原因を分析し (2) 再発防止のため該当ルールを修正・明確化する、の2点を自動的に実行すること。
